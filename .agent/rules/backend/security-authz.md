# 後端規則：安全與授權（AuthZ）

## 適用情境

- 新增需登入或權限控管的 API
- 調整角色、功能權限或存取邏輯
- 涉及敏感資料存取與管理操作

## 必做步驟

1. 先定義該功能需要的最小權限（least privilege）。
2. 在路由層加上對應 Guard/權限檢查，不只在前端限制。
3. 針對敏感操作明確區分 401（未登入）與 403（無權限）。
4. 確保審計需求：必要時保留操作紀錄或可追溯資訊。
5. 變更權限模型後，回頭驗證既有角色不會意外取得高權限。

## 禁止事項

- 不可只靠前端顯示控制來做授權。
- 不可在 service 內硬編碼散落的權限字串而無統一語意。
- 不可省略敏感欄位保護與輸出過濾。

## 驗證方式

- 未登入、權限不足、權限符合三種情境皆有正確回應。
- 敏感 endpoint 無法被低權限角色誤用。
- 權限調整後既有流程無回歸問題。

## 常見失誤

- Guard 掛在部分路由，漏掉同模組的其他寫入操作。
- 401/403 混用，造成除錯與使用者訊息混亂。
- 權限變更未同步更新文件與測試案例。
